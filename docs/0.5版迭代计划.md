# 0.5版迭代计划：稳定性与长停顿体验优化

## 1. 背景
0.4 版本在真实使用中暴露出两类高频问题：
1. 按住 `Option` 说话时，若中间有较长停顿，松键后只上屏前半段，后半段丢失。
2. 连续使用一段时间后，偶发“无法识别”，需要重启或等待恢复。

这两类问题都直接影响“可用性信任”，因此 0.5 版本目标是先把“稳定可用”做到可量化、可验证。

## 2. 0.5 目标（用户体验视角）
1. **不丢字**：长停顿场景下，整段语音最终都能上屏。
2. **不中断**：长时间连续使用时，识别链路可自恢复，不要求用户重启。
3. **可感知**：用户知道系统当前处于“录音中 / 收尾中 / 识别失败可重试”的哪个状态。

## 3. 版本范围与优先级

### P0（必须完成）
1. **会话收尾重构（解决长停顿丢字）**
   - 松键后不立即硬截断，进入 `FINALIZING_GRACE`（建议 1200ms，可配置）。
   - 收尾阶段继续接收并合并剩余 `FinalResult`，直到满足“静音结束”或超时。
   - 上屏内容来源改为“会话聚合器最终文本”，而不是“最后一次回调文本”。
2. **识别链路自愈（解决连续使用失效）**
   - 增加健康探针：连接状态、最后事件时间、队列积压、异常计数。
   - 超时或异常阈值触发“软重连”（重建识别流，不退出主进程）。
   - 连续失败触发“硬重置”（重建音频+识别组件），并给出可读提示。
3. **失败兜底与反馈**
   - 若收尾阶段无 Final 文本，自动回退到“最高置信度 Partial/缓存文本”。
   - 托盘和悬浮窗明确显示：`录音中`、`收尾中`、`重连中`、`失败可重试`。

### P1（建议完成）
1. 一键重试：最近一次识别失败后，快捷键双击触发“重试上一次会话”。
2. 故障快照：失败时自动记录最小诊断包（错误码、状态、最近 20 条事件）。
3. 可调参数面板：`收尾等待时长`、`静音阈值`、`重连阈值`。

## 4. 关键设计方案

## 4.1 会话状态机扩展
建议状态：
`IDLE -> RECORDING -> FINALIZING_GRACE -> RECOGNIZING_DRAIN -> PASTING -> IDLE`

设计要点：
1. `Option` 松开只表示“停止采集新音频”，不等于“立即完成识别”。
2. 在 `RECOGNIZING_DRAIN` 中等待识别服务吐完尾包（drain）。
3. 到达超时仍无新结果时，再统一上屏与清理资源。

## 4.2 文本聚合器（防丢字核心）
1. 每个会话维护 `segments[]`（按时间戳存储 partial/final）。
2. `FinalResult` 覆盖对应 segment，`PartialResult` 只作为临时展示。
3. 上屏前执行一次 `final_text = merge(segments)`，避免“只取最后回调”。

## 4.3 稳定性治理
1. **资源治理**：线程、音频流、WebSocket 连接都必须有显式生命周期和超时回收。
2. **幂等停止**：`stop_session()` 可重复调用，不产生脏状态。
3. **看门狗**：每 1s 检查一次“最后事件时间”和“状态停滞时长”。
4. **分级恢复**：先软重连，再硬重置，最后才提示用户手动干预。

## 5. 验收标准（0.5 新增）
1. **长停顿不丢字**：
   - 用例：按住说话 10s，中间停顿 2-3s，再继续说，松键后整段上屏。
   - 通过标准：30 次重复测试，文本完整率 100%。
2. **连续使用稳定性**：
   - 用例：连续 200 次短按 + 50 次长按。
   - 通过标准：会话成功率 >= 99%，无“必须重启应用”案例。
3. **恢复能力**：
   - 用例：识别中断网、服务异常、鉴权失败后恢复。
   - 通过标准：异常后 10 秒内恢复到可继续识别状态。
4. **状态可感知**：
   - 通过标准：所有关键中间态均有 UI 文案反馈，且文案与真实状态一致。

## 6. 里程碑
1. **M1（第 1 周）**：状态机与文本聚合器改造设计+单测。
2. **M2（第 2 周）**：收尾逻辑、drain 机制、分级恢复机制开发完成。
3. **M3（第 3 周）**：稳定性压测、日志与诊断补齐、灰度发布。

## 7. 发布策略
1. 先灰度给高频用户，默认开启新收尾策略。
2. 保留开关：可回退到 0.4 收尾路径（仅紧急止损使用）。
3. 观察 3 天核心指标（成功率、丢字反馈、重连次数）后全量。

## 8. 指标看板（建议）
1. `session_success_rate`
2. `missing_text_report_rate`
3. `auto_recover_success_rate`
4. `avg_finalize_latency_ms`
5. `forced_restart_count`

---

结论：0.5 不追求新功能堆叠，优先把“长停顿不丢字 + 长时使用不掉线”做到工程化闭环，再继续扩展体验功能。
